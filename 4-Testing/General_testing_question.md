# Load vs stress testing

## Load Testing
Primary Goal- Verify performance under expected traffic.
Load Level- Normal or anticipated peak load (e.g., 1,000 users).
Focus- Response time, throughput, and bottlenecks.
Analogy- Carrying a heavy backpack you're used to.

## stress testing 
Find the breaking point and recovery behavior.
Extreme load beyond capacity (e.g., 5,000 users).
Robustness, error handling, and system recovery.
Adding bricks until your back gives out.

## Spike Testing: 
A subset of stress testing that simulates a sudden, massive increase in traffic (like a flash sale starting at midnight).

## Soak (Endurance) Testing: 
Checking if the system can handle a sustained load over a long period (e.g., 24–48 hours) to find long-term memory leaks.

## Scalability Testing: 
Measuring how well the system "scales up" (adding more servers) as the load increases.


# DEBUGGING
1. Verification & Isolation (where ?)
Before diving into code or logs, define the boundaries of the bug.
-Reproduce Consistently: Find the exact "Minimum Reproducible Example." If a 10-step test fails, try to see if it still fails with only 3 steps.
- Cross-Browser/Device Check: Is it a Frontend (UI) bug specific to Safari or an iPhone 15? Or does it happen everywhere?
- Check if it is happening only locally, only on simulator/real device, when running through CI pipeline, when running through test automation.
- Classify the bug - UI or backend 
- If UI - Layout & Alignment, Static Content Errors:, Interaction Failures, Device-Specific Issues: , Animation Glitches: 
- If backend - "The Null/Empty State, Data Mismatches:, Infinite Loading: , Functional Errors on Submit:, Stale Data: 
- Check Logs - device or system logs, browser or appium logs, framework logs , backend logs
- verify the screenshots and artifacts generated by test

Once we have gathered data about the issue, isolated it we try to identify where the issue is happening to provide more info to the developer 


2. The "What Changed?" Audit: 
Check the build version. Was a new feature merged? Was there a configuration change in the environment?
Try to repro the bug with least variables.

3. Where the issue could be ?
- Account/Data Isolation/env isolation: if the issue is Data-specific (corrupt database record) or Logic-specific.
- App Crash / Freeze - Client-side (Console/Sentry)
"Something went wrong" Popup - Network Proxy (Proxyman)
Missing / Wrong Data- Backend Logs (ELK) + Database
Endless Loading Spinner - Network (Timeout) or API Gateway
Slow Performance - APM Logs (New Relic / Grafana)
- Toggle Feature Flags:
- Environmental & External Factors
Network Throttling: Test on 3G vs. Wi-Fi. Some bugs only appear during high latency.
Cache & Persistence: Clear cookies, local storage, and app cache. Does the bug disappear? If so, the RCA is State Persistence.

# If the issue happening when running test from automation - Frontend & UI automation framework Debugging
If the failure is visible on the screen, start here.
Step 1: The Visual Check: Review the screenshot or video recording from the failure.
Is the element there? If yes, it’s a synchronization/wait issue.
Is the app on the wrong screen? The previous test might have left "dirty data."
Step 2: Inspect the Hierarchy: Use the Appium Inspector or Xcode View Debugger.
Check if the accessibilityId or label has changed. In iOS, developers sometimes change the UI structure, which breaks xpath or iOSClassChain locators.
Step 3: Driver Logs: Check the WebDriverAgent (WDA) logs.
If you see StaleElementReferenceException, the UI refreshed while you were interacting.
If you see SessionNotCreated, the WDA server on the iPhone crashed.
Step 4: The "Quiescence" Check: In iOS, the driver often hangs because it's waiting for an animation to finish. If the app is "busy," the click won't fire.
Identify "State Contamination"
CI often runs tests in parallel or in a specific sequence that differs from your local execution.
Dirty Data: Does the test depend on a user account that was altered by a previous test in the CI pipeline?
Solution: Use the "Clean State" strategy.
Java

```
    XCUITestOptions options = new XCUITestOptions();
    options.setFullReset(true); // Ensures a fresh install on every CI run
    options.setShouldTerminateApp(true);
```

# If the issue looks to be coming from backend - Network & API Layer (The "Middleman")
Network Layer (The "Source of Truth"): I use Proxyman or Charles Proxy to inspect the handshake.
Request: Did the UI send the right parameters?
Response: Did the API return a 200 with "empty data" or a 500 with a stack trace? 

# Backend & Server-Side Analysis
If the API returns a 500 Internal Server Error, the RCA is in the backend.
Log Peeking (Splunk/ELK/Datadog/Azure Monitor Application Insights.): Search for the trace-id or correlation-id from the API response. Look for "Stack Traces."
Database Querying: If the UI shows "No Data," query the DB directly. If the data is in the DB but not on the screen, the bug is in the API Mapping layer.
Microservices Dependency: Check if the service you are testing is failing because a different downstream service is down.

# Debugging API Test Frameworks (Rest Assured)
API debugging is more logical and data-driven. It usually boils down to Contract vs. Payload.
Step 1: Status Code Analysis: * 4xx: Your request is bad (Check Auth tokens, Headers, or JSON body).
5xx: The server is failing (Check backend logs or DB connectivity).
Step 2: Log Interception: Use Rest Assured’s logging capability to see exactly what was sent.
Java
```
    // Non-fluent logging example
    RequestSpecification request = RestAssured.given();
    request.log().all(); // Logs Headers, Body, Params
    Response response = request.post("/endpoint");
    response.log().all(); // Logs the Response JSON
```



Step 3: Schema Validation: If the test fails on an assertion, check the JSON Schema. Did the developer change a field from an Integer to a String?
Step 4: Manual Verification: Copy the failing payload into Postman or cURL. If it fails there, it’s a backend bug. If it passes, the issue is in your Java framework's serialization (Gson/Jackson).

# Debugging in CI (Jenkins/GitHub Actions)
CI debugging is about Environmental Deltas. The "It works on my machine" problem.
Step 1: Resource Contention: CI agents often have less CPU/RAM. Check if the failure is due to a Timeout.
Fix: Increase the WebDriverWait for CI runs.
Step 2: Artifact Analysis: Download the Allure Report or the console-output.log.
Look for network "jitter." If the CI agent is in a different region than the staging server, latency can cause tests to fail.
Step 3: Headless Mode Issues: CI runs simulators in "headless" mode. Sometimes, rendering issues occur in the background that don't happen when the simulator is visible on your laptop.
Step 4: Parallelization Collisions: If you run 50 tests at once, are two tests trying to use the same "Test User"? This causes 409 Conflict errors.

# "How do you debug an intermittent (flaky) bug?"

1. run it several times to know how intermittent the bug is 
Reproduce the bug in different ios versions/sizes/environments to rule if it is device or env dependent bug.
Reproduce the bug locally to rule out if it is CI dependent, automation framework dependent or dirty data.
2. Analyze Logs, Screenshots, and Reports I always check the available logs, screenshots, or video recordings captured during test execution. For UI tests, tools like Selenium or Playwright often provide detailed driver logs that help identify timing or synchronization issues. If API tests are failing intermittently, I examine request/response payloads, server logs, and any rate-limiting or timeout-related behavior. Collecting evidence across multiple failing runs helps me notice patterns such as specific UI elements not being ready, environment instability, or services responding inconsistently.

Unreliable 3rd party integration API causing failure intermittently. 
3. Investigate Common Flakiness Causes From experience, flaky tests usually fall into a few categories:
Timing/Sync Issues: The test interacts with elements before they are ready (e.g., not using explicit waits).
Test Data Problems: Reuse of the same test data leading to conflicts, duplication, or unexpected state.
Environment/Infrastructure Issues: Network latency, environment downtime, or unstable third-party dependencies.
Parallel Execution Conflicts: Tests interfering with each other when running concurrently.
Assertions Too Strict or Fragile: Tests failing due to minor, non-functional differences (e.g., checking for exact text instead of a pattern).

4. Add Instrumentation and Diagnostics If the root cause isn’t immediately obvious, I add extra logging, screenshots at critical steps, or debug-specific waits to capture more details during execution. In distributed CI/CD pipelines, I also look at machine configurations (e.g., memory usage, browser version mismatches, or container scaling issues). These diagnostics help confirm whether the problem lies in the test code, the framework, or the system under test.
5. Apply Fixes and Improvements Once the root cause is identified, I fix the underlying issue rather than masking it. For example:
For synchronization issues, I replace hardcoded sleeps with explicit waits or smarter polling mechanisms.
For unstable test data, I create unique data per run or implement better cleanup strategies.
For parallel conflicts, I ensure tests are isolated (using mock services, test containers, or dedicated data).
For fragile assertions, I make them more robust and aligned with actual business expectations.
6. Validate the Fix After applying a fix, I re-run the test multiple times under different environments (local, staging, CI/CD) to ensure stability. If possible, I also use test rerun strategies or quarantine mechanisms temporarily until I confirm the fix holds over time.
7. Prevent Future Flakiness Finally, I treat flaky tests as a learning opportunity. I share the root cause and resolution with the team, add coding guidelines (e.g., always use explicit waits for UI tests, avoid shared state), and incorporate checks into the framework to reduce similar issues. For critical pipelines, I sometimes add retry mechanisms with proper reporting, but only as a fallback, not a replacement for fixing the real problem.

# "What do you do if you can't reproduce a bug found in Production?"
Answer: "I look at Observability data. I check the production logs (Splunk/ELK) using the trace-id or correlation-id from the reported error. I also check the Device Metadata (iOS version, hardware model, battery level, network type). Often, 'unreproducible' bugs are specific to an older iPhone model or a specific 'Low Power Mode' state."

# "A developer says 'Works on my machine.' How do you respond?"
Answer: "I don't argue; I compare the environment. I ask the developer for their build commit hash, their simulator version, and their local config. I then offer to Pair Debug. I run my failing script on their machine. If it passes there, we know the issue is the Test Infrastructure. If it fails there, the developer can see the failure happening in their own debugger, which speeds up the fix."
Compare Environments: Check Node/Java versions, OS, and Library versions.
Check the 'Cleanliness' of the build: Ensure they aren't relying on a local cache or 'poisoned' data that makes the bug disappear.
Review the CI Pipeline: If it works for them but fails in CI, the issue is likely a Race Condition or a missing Environment Variable in our pipeline.

# estimations

1. Define the Release Scope and Requirements
Analyze User Stories: Read all stories to understand acceptance criteria, UX flows, and business intent. If requirements are vague, stop and clarify with the PO/BA.
Identify Testing Types: Determine which tests are required: Functional, Regression, Performance, Security, or UAT.
Define Coverage Matrix: List devices, operating systems, and browsers needed for testing, using market share or user analytics to guide the strategy. [1, 4, 6, 7]
2. Decompose the Work (Work Breakdown Structure - WBS) [4, 5, 8]

Break testing down into smaller, manageable tasks. A typical breakdown includes:
Test Design: Time to create test cases or update BDD scenarios.
Test Data/Environment Setup: Provisioning environments and preparing necessary data.
Manual Execution: Running happy path + edge cases.
Automation: Developing new scripts or updating existing ones.
Bug Triage/Retest: Time for retesting bugs after fixes.
Regression Suite Run: Ensuring old functionality remains stable. [4, 10, 11, 12]
3. Choose an Estimation Method
Three-Point Estimation (PERT): Use this for uncertainty. Calculate: (Optimistic + $4 \times$ Most Likely + Pessimistic) $\div$ 6.
Percentage of Development Time: A common shortcut is to take a percentage of the total development estimation (e.g., 25%–50% of dev time), though this should be verified with a custom estimate.
Historical Data: Use data from similar past projects. [1, 3, 13, 14]
4. Factor in Risks and Buffers
Environment Stability: If the staging environment is often broken, add a buffer.
Third-Party Dependencies: If testing depends on third-party APIs that may be slow, add extra time.
Rework/Bugs: Allocate 20–30% of total testing time for exploratory testing and unforeseen bugs. [3, 4]
5. Validate and Refine Estimates
Peer Review: Walk through the estimate with other QA team members.
Team Consensus: Discuss estimates with developers to ensure the technical complexity matches your assumptions.
Re-estimate if Needed: If the scope changes during the sprint, recalculate and communicate the new timeline immediately.

- Building an "intelligent retry" or "self-healing" system
Building an "intelligent retry" or "self-healing" system is a significant step forward for a Senior SDET. Since you're already using **GitHub Copilot** and **LangChain**, you have the right foundation to move from simple conditional retries to an AI-driven agentic approach.

Here is how you can build an intelligent retry system that finds fallback locators at runtime using AI agents.

---

### 1. The "Agentic" Self-Healing Architecture

Instead of a hard-coded list of selectors, you can implement a **Heuristic Agent**. When a standard `findElement` fails, the agent takes over to "repair" the locator.

* **Trigger:** Catch `NoSuchElementException` or `TimeoutException`.
* **The Agent (LangChain):** Create a small utility that takes the current DOM snippet and the "broken" locator as input.
* **The Prompt:** Ask the LLM: *"The locator `#login-btn` failed. Based on this HTML snippet, what is the most likely new CSS selector for the login button? Provide only the selector."*

### 2. Implementation Steps

#### Step A: Capture Context on Failure

To help an AI find a fallback, it needs to see what the page looks like *now*.

```java
// Conceptual Java example for your framework
public WebElement findSmartElement(By primaryLocator) {
    try {
        return driver.findElement(primaryLocator);
    } catch (NoSuchElementException e) {
        // 1. Get the HTML of the parent container or the whole page
        String htmlContext = driver.getPageSource(); 
        
        // 2. Call your AI Agent (via LangChain or an internal API)
        String fallbackSelector = AIAgent.getFallback(primaryLocator.toString(), htmlContext);
        
        // 3. Retry with the new selector
        return driver.findElement(By.cssSelector(fallbackSelector));
    }
}

```

#### Step B: Use Copilot to Write the Heuristics

You can use **GitHub Copilot** to generate a "Probability Map" of locators. Instead of one ID, Copilot can help you write a utility that generates a ranked list of attributes (`data-testid`, `aria-label`, `name`, `class`) to try in sequence.

> **Copilot Prompt:** *"Generate a Java method that returns a list of potential fallback By locators for an element based on its known attributes like 'submit-button' ID and 'Login' text."*

### 3. Integrated Strategies for Your Framework

| Strategy | Tool/Tech | How it works |
| --- | --- | --- |
| **Fuzzy Matching** | Selenium 4 + AI | Uses [Relative Locators](https://www.google.com/search?q=https://www.selenium.dev/documentation/webdriver/elements/relative_locators/) to find elements "near" a stable label if the exact ID changed. |
| **DOM Snapshotting** | LangChain | Passes the failed locator and a `DOM` tree to an LLM to predict the most likely "moved" element. |
| **Auto-update** | GitHub Actions | If a fallback works, the agent creates a **Pull Request** or logs a "Healed" status to update your POM file automatically. |

### 4. Advanced: The "Agent" as a Quality Gate

Since you are already working with **AI-driven agents and webhooks**, you can extend this:

1. **Failure Event:** Your test suite sends a webhook to your LangChain agent when a locator fails.
2. **Analysis:** The agent analyzes the `DOM` and the last known good screenshot.
3. **Correction:** The agent provides a "Suggested Fix" directly in your **Jira** ticket or **Slack** channel, saving you the manual triage time.

---

### Pro-Tip for Senior SDETs

**Don't send the entire `DOM` to an LLM.** It's expensive and slow. Use a utility to strip the HTML down to just relevant tags (buttons, inputs, links) and their immediate parents before sending it to your "Smart Retry" agent.

# load testing

To effectively implement load testing—whether you're using the [JMeter Java DSL](https://jmeter.apache.org/usermanual/build-programmatic-test-plan.html) or traditional [REST Assured performance assertions](https://medium.com/javarevisited/rest-api-load-performance-testing-using-apache-jmeter-63605572e862)—you need to understand both the "language" of performance and how to structure your test cases.

---

### 1. Key Performance Terminology

As an SDET, you'll use these metrics to determine if a build passes or fails.

| Term | Definition | SDET Context |
| --- | --- | --- |
| **Throughput** | The number of requests a system handles per unit of time (e.g., Requests Per Second - RPS). | High throughput with low errors is the goal. |
| **Response Time** | The total time taken from sending a request to receiving the full response. | Usually measured in 90th or 95th percentiles ($P90$/$P95$). |
| **Latency** | The delay before a transfer of data begins following an instruction for its transfer. | Often called "Time to First Byte." |
| **Ramp-Up** | The time it takes to get the full number of virtual users (Threads) running. | Prevents "system shock" by adding users gradually. |
| **Think Time** | The simulated delay between a user's actions. | Use `Thread.sleep()` or JMeter Timers to simulate real user behavior. |
| **SLA (Service Level Agreement)** | The predefined performance contract. | Example: "Login must complete in < 2 seconds for 500 concurrent users." |

---

### 2. Sample Load Test Cases

When designing tests for your [REST Assured](https://medium.com/javarevisited/rest-api-load-performance-testing-using-apache-jmeter-63605572e862) framework, focus on these common scenarios:

#### **Case 1: The Baseline (Happy Path)**

* **Goal:** Determine how the system performs under "normal" expected load.
* **Scenario:** Run 50 concurrent users for 10 minutes.
* **Success Criteria:** $P95$ Response Time < 500ms; 0% Error Rate.

#### **Case 2: The Peak Load (Stress Test)**

* **Goal:** Find the "breaking point" where the system starts failing or slows down significantly.
* **Scenario:** Start with 10 users and increase by 10 every minute until the error rate exceeds 5%.
* **Success Criteria:** Identify the maximum RPS the system can handle.

#### **Case 3: Endurance (Soak Test)**

* **Goal:** Check for memory leaks or resource exhaustion over time.
* **Scenario:** Run 75% of peak load for 4–8 hours.
* **Success Criteria:** Memory usage remains stable; no "Slow Memory Leaks" detected.

#### **Case 4: Spike Test**

* **Goal:** Test how the system handles a sudden, massive influx of users (e.g., a "Flash Sale").
* **Scenario:** Jump from 10 users to 500 users in 5 seconds.
* **Success Criteria:** System recovers quickly once the spike subsides.

---

### 3. Implementation Example: Asserting in Java

Using the [JMeter Java DSL](https://jmeter.apache.org/usermanual/build-programmatic-test-plan.html) makes these test cases readable as code:

```java
@Test
public void loadTestSLA() throws IOException {
    TestPlanStats stats = testPlan(
        threadGroup(100, Duration.ofMinutes(1), // 100 users, 1 min ramp
            httpSampler("https://api.myapp.com/login")
                .header("Content-Type", "application/json")
        )
    ).run();

    // Terminology in Action: Asserting against P95
    assertTrue(stats.overall().sampleTime().percentile95().toMillis() < 800, 
               "Login is too slow under load!");
}

```
